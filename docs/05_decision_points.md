# 05 — نقاط الحسم (يسألها Claude Code أثناء التنفيذ)

> **القاعدة:** لا تفترض أياً من هذه القرارات. عند بلوغ النقطة المعنية في البناء، توقّف، اطرح السؤال على المستخدم (وليد)، انتظر الإجابة، ثم تابع. سجّل الإجابة في هذا الملف لاحقاً للرجوع إليها.

| # | متى تسأل | السؤال | لماذا يهم |
|---|---|---|---|
| 1 | عند بناء اعتماد التعميد (`rentals`) | اعتماد واحد أم سلسلة اعتمادات (نقليات ثم مدير أصول)؟ | يحدّد آلة حالة الطلب وعدد خطوات الموافقة |
| 2 | عند بناء الصيانة الوقائية (`maintenance`) | هل أربط عدّاد التشغيل/الكيلومترات بجدولة الصيانة (كل X ساعة/كم)؟ | يضيف حقول عدّاد ومهام مجدولة |
| 3 | عند بناء التاب المالي/الإهلاك (`assets`) | الإهلاك يُحتسب داخل المنصة أم يُجلب من نظام محاسبي خارجي؟ | تكامل محاسبي أم منطق إهلاك داخلي |
| 4 | عند بناء الاستلام/التسليم (`handover`) | توقيع إلكتروني رسمي أم تأكيد داخلي يكفي؟ | يحدّد الحاجة لمزوّد توقيع وحقول إثبات |
| 5 | عند بناء البيع (`disposal`) | اعتماد واحد (مدير أصول) أم اعتماد مالي إضافي قبل خصم الأصل؟ | عدد خطوات اعتماد `SaleOrder` |
| 6 | عند بناء الاستحواذ الخارجي (`acquisition`) | من يتحمّل صيانة/تأمين الأصل المستأجَر (الشركة أم المورّد)؟ | يحدّد احتساب OpEx/TCO وقيم `maintenanceBearer`/`insuranceBearer` |

## سجل الإجابات (يملؤه Claude Code بعد سؤال المستخدم)
- [x] #1: **اعتماد واحد (النقليات)** — مدير المشروع يطلب ← النقليات تراجع التوفّر وتعتمد/ترفض ← يُصدر التعميد والعقد. آلة الحالة: PENDING → APPROVED (حجز أصل) → FULFILLED (عقد ساري)، أو REJECTED/CANCELLED. [أُجيب 2026-06-26]
- [x] #2: **لا ربط بالعدّاد الآن** — بوابة الصيانة تعتمد أوامر الصيانة (تصحيحية/وقائية كنوع فقط) + الكرت + الفواتير + الإغلاق وترحيل التكلفة، دون تتبّع عدّاد تشغيل/كيلومترات أو جدولة آلية. يُترك للمراحل اللاحقة. [أُجيب 2026-06-26]
- [x] #3: **هجين** — المنصة تحسب الإهلاك والقيمة الدفترية تلقائياً (قسط ثابت من سعر الشراء + نسبة الإهلاك + تاريخ الشراء)، **وفي نفس الوقت** يمكن للمستخدم إدخال القيمة وتعديلها يدوياً (تجاوز يدوي override). التاب المالي يعرض القيمة المحسوبة والقيمة المُدخلة معاً. [أُجيب 2026-06-26]
- [x] #4: **تأكيد داخلي موثّق** — لا مزوّد توقيع خارجي. فحص الاستلام/التسليم يُثبّت اسم الموقّع ودوره وختماً زمنياً وIP وصوراً، ضمن سجل تدقيق غير قابل للتعديل. [أُجيب 2026-06-27]
- [x] #5: **اعتماد واحد (فصل مهام)** — مدير الأصول يقترح أمر البيع، ومعتمِد مختلف يعتمده، ثم يُعرض للبيع ثم يُباع/يُسحب. من يقترح ≠ من يعتمد. [أُجيب 2026-06-27]
- [x] #6: **يُحدّد لكل عقد** — عند إنشاء عقد الاستئجار الخارجي يُختار متحمّل الصيانة والتأمين (شركة/مورّد) منفصلين؛ ما تتحمّله الشركة فقط يدخل OpEx/TCO. [أُجيب 2026-06-27]

> **الحالة (2026-06-28):** نقاط الحسم الستّ **كلها محسومة** ومنفّذة (أعلاه).

## قرارات طبقة الـ SaaS (لم تكن نقاط حسم أصلية — حُسمت بالتشاور)
- [x] **الفوترة:** محفظة داخلية أولًا، ثم **بوابة دفع Tap** حقيقية (شحن/مقاعد/إضافات بالبطاقة).
- [x] **إسناد النقليات:** خطوة واحدة — النقليات تُسند الأصل وتشغّله مباشرة (يُذكر المشروع).
- [x] **التخزين:** حساب مشترك للمنصّة + خيار bucket مخصّص لكل شركة (BYO)، بمجلد معزول `tenant_<id>/`.
- [x] **مشغّل المنصّة:** هوية في **جدول مستقل** (`PlatformAdmin`) لعزل أثر الاختراق.
- [x] **الأدوار:** معزولة لكل شركة (`Role`/`RolePermission` بـ `tenantId`).
- [x] **النطاقات الفرعية (subdomains):** مؤجَّلة — تحديد المستأجر من الـ JWT لا من الرابط (الأسلم).
- [x] **تطبيق السائقين:** مؤجَّل مؤقتًا باختيار المستخدم (CLAUDE.md §6)، وأساسه مُهيّأ.
- [x] **الفوترة الخارجية/ZATCA:** **خارج النطاق** (التأجير داخلي = تخصيص تكلفة، لا فوترة عملاء).

> التفصيل الكامل لما نُفّذ في [`06_implemented_state.md`](06_implemented_state.md).

## نقاط حسم رؤية الـ SaaS الموسّعة (أُجيبت 2026-06-28)
- [x] **(1) العزل:** **مشترك + PostgreSQL RLS للجميع** كطبقة دفاع ثانية فوق scoping القائم،
  مع إبقاء باب ترقية Enterprise (DB/Bucket مستقل) مفتوحًا مستقبلًا (التخزين المخصّص موجود أصلًا).
- [x] **(2) سياسة تجاوز التخزين:** **حدّ ليّن + تنبيه عند الاقتراب، ثم منع الرفع الجديد عند 100%** (قابل للضبط من المنصّة).
- [x] **(3) مزوّد الدفع:** Tap مدمج؛ **إبقاء الواجهة مجرّدة وإضافة مزوّد ثانٍ** (تجريد Provider-agnostic كامل).
- [x] **حقول حساب الشركة:** تُبنى الآن (البريد/الهاتف/المدينة/السجل التجاري/الرقم الضريبي + الاسم النظامي) — تخدم أيضًا حقول «المشتري» في فاتورة المرحلة 5.
- [x] **المرحلة 1 (الجزء الآمن) نُفّذت:** `TenantUsage` (لقطة مقاعد/تخزين تُحدَّث آليًا) + حالة `CANCELED`
  للمستأجر (دخول مُغلق + إعادة تفعيل من المنصّة). تم التحقّق.
- [x] **(1-RLS) القرار: تأجيل RLS مع توثيق خطة التفعيل.** السبب: الاتصال الحالي بدور `postgres`
  (superuser **يتجاوز RLS**)، والتفعيل الحقيقي يتطلّب دور تطبيق مقيّد + ضبط GUC لكل طلب + تعديل ٦ مواضع
  `$transaction` — تعقيد/مخاطرة على منصّة لايف. عزل طبقة التطبيق (Prisma middleware) مفعّل ومُختبَر 7/7
  ويبقى الطبقة المفروضة. **خطة التفعيل موثّقة في [`06_implemented_state.md`](06_implemented_state.md#ملحق-تفعيل-rls).**
- [x] **(4) assetCap:** يُضاف كحقل في الباقة/الاشتراك **دون فرضه الآن** (قابل للتفعيل لاحقًا).
- [x] **(5) تردد الإحداثيات/الاحتفاظ:** قابل للضبط من الجهاز (افتراض ~30 ثانية)؛ تُحفظ كل الإشارات الخام
  (`LocationPing` مفهرس `assetId, recordedAt`)؛ الأرشفة/التجزئة عند الحجم العالي مؤجَّلة كتحسين موثّق.
- [x] **(6) تأمين endpoint الأجهزة:** **HMAC-SHA256 لكل جهاز** عبر `signingKey` (رأس `x-signature` فوق
  `externalId|lat|lng|recordedAt`)؛ توقيع خاطئ → 401.
- [x] **(7) احتساب المركبة في الحصة:** **عند التفعيل** (Asset.trackingEnabled / تسجيل الجهاز)، لا عند أول ping.

## اكتمال المراحل (2026-06-28)
المراحل **0→7 كلها منفّذة ومتحقَّقة**: حقول الشركة · Tenancy(Usage/Cancel) · الباقات · حصص R2 ·
**الفاتورة الضريبية** · **تتبّع المركبات** · **طلب ربط وصل**. التفاصيل في [`06_implemented_state.md`](06_implemented_state.md).
