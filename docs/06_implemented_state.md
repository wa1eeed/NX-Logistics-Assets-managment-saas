# 06 — الحالة المنفّذة (As-Built) — مرجع شامل

> هذا المستند هو **مصدر الحقيقة لما تم بناؤه فعلاً** حتى 2026-06-28. الوثائق 01–05 هي
> مواصفات البناء الأصلية (الـ Kit)؛ هذا الملف يوثّق ما نُفّذ ووُسّع وصُحّح فوقها —
> وأهمّه تحويل المنصّة من نظام أحادي الشركة إلى **منصّة SaaS متعدّدة المستأجرين**.

اسم المنتج: **NX-LAM** — منصّة إدارة وتشغيل الأصول (مركبات ومعدات) كخدمة SaaS.

---

## 0) نظرة عامة على ما أُنجز

بُني الأساس الوظيفي بالكامل (الأصول، النقليات/التأجير، الصيانة، البيع، الاستحواذ،
السائقون، المؤشرات، الاستلام/التسليم، التدقيق، البيانات المرجعية، الإدارات/الوحدات)
ثم أُضيفت فوقه **طبقة SaaS كاملة**:

1. **العزل متعدّد المستأجرين** (Multi-tenant isolation) عبر `tenantId` + وسيط Prisma.
2. **محرّك الاستحقاقات** (Entitlements): سقوف تخزين/مستخدمين + تفعيل موديولات لكل باقة.
3. **الفوترة الذاتية + المحفظة** (Billing/Wallet): شراء مقاعد وتفعيل إضافات.
4. **بوابة دفع حقيقية — Tap**: شحن المحفظة/المقاعد/الإضافات بالبطاقة (Hosted Checkout + Webhook + Sandbox).
5. **لوحة مشغّل المنصّة** (Platform Admin) فوق كل الشركات، بهوية معزولة في جدول مستقل.
6. **التسجيل الذاتي** للشركات + **الهوية البصرية** لكل شركة (اسم/شعار/لون).
7. **الأدوار والصلاحيات معزولة لكل شركة** (Tenant-scoped roles).
8. **استيراد الأسطول الحقيقي** لأول شركة (1947 أصلاً من ملف العميل) + حقول مخصّصة لكل نوع.
9. **الصيانة الوقائية + لوحة الامتثال** (عدّادات/كيلومترات + جداول وقائية + تنبيهات الوثائق).

> **خارج النطاق صراحةً:** الفوترة الخارجية/ZATCA (التأجير داخلي = تخصيص تكلفة، لا فوترة عملاء — راجع 01 §التأجير الداخلي). و**تطبيق السائقين** (React Native) مؤجَّل عمداً (CLAUDE.md §6) وأساسه مُهيّأ.

## 1) المعمارية متعدّدة المستأجرين (Multi-Tenancy)

النمط المعتمد: **قاعدة بيانات واحدة مشتركة + عمود `tenantId` على كل كيان جذري + عزل
على مستوى الاستعلام** (Shared DB, row-level isolation) — وهو الأنسب معياريًا دون تعقيد غير ضروري.

- **سياق المستأجر:** `AsyncLocalStorage` (`tenant-context.ts`) يحمل `tenantId` لكل طلب.
- **الحقن التلقائي:** `TenantContextInterceptor` يضبط السياق من `req.user.tenantId`.
- **العزل المحكم:** وسيط Prisma (`$use` في `prisma.service.ts`) يضيف فلتر `tenantId`
  تلقائياً على كل قراءة/تحديث/حذف، ويحقن `tenantId` على كل إنشاء، لكل الموديلات في
  `TENANT_MODELS`. غياب السياق (seed، مسارات عامة، مشغّل المنصّة) ⇒ لا عزل (مقصود).
- **الحراس قبل المعترِضات:** `ModuleAccessGuard` يقرأ `req.user.tenantId` مباشرة لأن
  الحراس تعمل قبل ضبط سياق الـ ALS.
- **التحقق:** اختبارات عزل 7/7؛ كل شركة ترى بياناتها فقط (مثال: NX ترى 1947 أصلاً،
  وشركة مسجَّلة حديثاً ترى 0).

**فصل مشغّل المنصّة (Control-Plane):** مدير المنصّة يعيش في جدول **`PlatformAdmin`
مستقل تماماً** (ليس صفًّا في جدول مستخدمي الشركات) لعزل أثر أي اختراق على جانب
المستأجر عن صلاحيات المنصّة. مصادقة ثنائية الهوية عبر مطالبة `kind: 'tenant' | 'platform'`
في الـ JWT؛ تُحلّ الصلاحيات من الجدول الصحيح حسب النوع. `tenantId = null` للمنصّة ⇒
لا عزل ⇒ ترى كل الشركات (مع تمرير `tenantId` صريح في كل استعلام منصّة).

## 2) موديولات الـ API (NestJS)

`apps/api/src/modules/`:

| النطاق الأساسي | موديولات SaaS |
|---|---|
| `auth` (JWT + register), `rbac` (roles/permissions), `users`, `org-units` | `entitlements` (سقوف/حُرّاس) |
| `assets`, `asset-types`, `catalog` (أصناف + ماركة←موديل) | `billing` (محفظة/مقاعد/إضافات) |
| `rentals`, `handover`, `maintenance`, `preventive` | `payments` (بوابة Tap) |
| `disposal`, `acquisition`, `suppliers`, `drivers` | `platform` (مشغّل المنصّة) |
| `kpis`, `dashboard`, `lookups`, `settings`, `audit`, `notifications` | `tenant` (هوية/شعار الشركة) |

تكاملات: `integrations/storage` (S3-متوافق) و`integrations/payments` (Tap).

الطبقات: Controller (HTTP) → Service (منطق) → Prisma (بيانات) → DTO (تحقّق class-validator).
حُرّاس عامون: `JwtAuthGuard` + `PermissionsGuard` + `ModuleAccessGuard`. معترِض تدقيق عام.

## 3) طبقة SaaS بالتفصيل

### 3.1 الاستحقاقات (Entitlements)
- `TenantSubscription` لكل شركة: `planName`, `status`, `maxUserCount`, `maxStorageBytes`,
  `enabledModules` (JSON)، `seatPriceMonthly`, `walletBalance`.
- حُرّاس 403 عند تجاوز السقوف: إضافة مستخدم فوق الحد، رفع ملف يتجاوز سعة التخزين،
  الوصول لموديول غير مفعّل (`@RequiresModule`).
- **كتالوج باقات `Plan` في القاعدة** يديره مشغّل المنصّة (إنشاء/تعديل: مقاعد، تخزين، سعر شهري،
  **سعر تتبّع المركبة `perVehiclePrice`**، **سقف أصول `assetCap` اختياري — غير مفروض حاليًا**، موديولات).
  `POST /platform/tenants/:id/apply-plan` يطبّق الباقة على الشركة (لقطة caps/features/price على الاشتراك).
  الاشتراك يحمل فترة الفوترة (`currentPeriodStart/End`, `renewsAt`). seed: 4 باقات STARTER/STANDARD/BUSINESS/ENTERPRISE.

### 3.2 الفوترة والمحفظة (Billing)
- محفظة مدفوعة مسبقاً لكل شركة؛ شراء **مقاعد إضافية** (يرفع `maxUserCount`) و**تفعيل
  إضافات موديولات** (يقلب علم الموديول) بالخصم من المحفظة.
- سجل حركات `WalletTransaction` (TOPUP / PURCHASE_SEATS / PURCHASE_MODULE).
- المسارات: `GET /billing` · `POST /billing/purchase/seats` · `POST /billing/purchase/module`.
  (شحن المحفظة المجاني أُزيل — التمويل الآن بأموال حقيقية عبر Tap فقط.)

### 3.3 بوابة الدفع — Tap
- **حساب تاجر واحد للمنصّة** يحصّل من الشركات. الإعدادات في `PlatformSetting` (key
  `integrations.tap`)، يحرّرها **مدير المنصّة فقط** (صلاحية `payments.manage`)؛ المفتاح
  السرّي لا يُعاد للواجهة أبداً (يُعرض `secretSet` فقط).
- **التدفّق (Hosted Checkout):** إنشاء `PaymentIntent` (PENDING) + شحنة Tap → تحويل
  المتصفّح لصفحة Tap → عودة إلى `/billing/return` + **Webhook** → **التحقق من الشحنة من
  Tap نفسه** (لا يُوثَق بالـ body) → **تطبيق مرة واحدة** (idempotent: قلب الحالة ذرّيًا
  PENDING→PAID قبل التطبيق).
- ثلاثة أغراض: `WALLET_TOPUP`, `SEATS`, `MODULE` — كلها قابلة للدفع بالبطاقة.
- **Sandbox للتطوير:** عند غياب مفتاح حقيقي يعمل وضع تجريبي (تأكيد محاكى)، ويُرفض
  تلقائياً بمجرد ضبط مفتاح مباشر (لا يمنح رصيدًا مجانيًا في الإنتاج).
- المسارات: `GET/PUT /payments/config`, `POST /payments/config/test` (منصّة) ·
  `POST /payments/checkout`, `GET /payments/verify/:id`, `GET /payments` (شركة) ·
  `@Public POST /payments/webhook`.

### 3.4 لوحة مشغّل المنصّة (Platform Admin)
- `GET /platform/overview` (كل الشركات + الاستهلاك + MRR تقديري) · إنشاء/تعليق شركة ·
  تعديل الاشتراك · **انتحال (impersonate)** للدعم · نشاط عابر للشركات.
- صلاحيات محجوزة (`PLATFORM_RESERVED_PERMISSIONS`): `platform.tenants.read/manage`,
  `platform.impersonate`, `entitlements.manage`, `payments.manage` — مستبعدة من `'*'`
  الخاص بسوبر أدمن الشركة فلا يرفع حدوده بنفسه.

### 3.5 التسجيل الذاتي + الهوية البصرية + بيانات الشركة
- `@Public POST /auth/register`: تُنشئ شركة معزولة (حالة TRIAL) + اشتراك افتراضي +
  أول SUPER_ADMIN + **دخول تلقائي**. صفحة `/register` عامة (وتقبل حقول الشركة اختياريًا).
- **الهوية لكل شركة** (`Tenant.brandName/primaryColor/logoKey`): اسم/لون/شعار تُضبط من
  الإعدادات وتُلوّن الواجهة كاملة (الشعار + الاسم + متغيّر `--primary`).
- **بيانات حساب الشركة** (`Tenant.legalName/email/contactPhone/city/crNumber/vatNumber`):
  الاسم النظامي + البريد + رقم التواصل + المدينة + السجل التجاري (١٠ أرقام) + الرقم الضريبي
  (١٥ رقمًا) — تُضبط من كرت «بيانات الشركة» في الإعدادات، وهي نفسها **حقول «المشتري» في الفاتورة الضريبية**.
- المسارات: `GET /tenant/me`, `PUT /tenant/branding`, `POST /tenant/logo`, `PUT /tenant/profile`.

### 3.6 الفاتورة الضريبية (Phase 5)
- نموذج `Invoice` + `InvoiceLine` بكامل حقول الفاتورة الضريبية (بائع/مشتري لقطة، تسلسل عام
  `INV-YYYY-NNNN`، VAT %15 شامل، بنود، حقول ZATCA محجوزة فارغة) — **بلا تكامل ZATCA**.
- **تُصدَر تلقائيًا** عند تأكيد أي دفعة Tap (`PaymentsService.applyOnce → InvoicesService.issueForPayment`).
  البائع = بيانات المنصّة (تُضبط من لوحة المنصّة)؛ المشتري = بيانات الشركة (المرحلة 0).
- المسارات: `GET /invoices` + `/invoices/mine/:id` (الشركة) · `GET/PUT /invoices/seller` (المنصّة) ·
  `GET /invoices/all`. الواجهة: قائمة فواتير + عارض قابل للطباعة (PDF عبر طباعة المتصفّح) + كرت بيانات البائع.

### 3.7 تتبّع المركبات (Phase 6)
- نماذج: `TrackingAddon` (حصة per-vehicle) · `TrackingDevice` (HARDWARE/DRIVER_APP/WASL + `signingKey`) ·
  `LocationPing` · `Geofence` + `Asset.trackingEnabled`. التفعيل = إضافة نشطة (باقة أو شراء مستقل).
- **التفعيل بالدفع:** غرض دفع `TRACKING` (عدد المركبات × `perVehiclePrice`) عبر Tap → يُنشئ `TrackingAddon`
  + فاتورة per-vehicle. **الحصة مفروضة عند التفعيل** (تفعيل أصل فوق الحصة → 403).
- **استقبال المواقع (Public):** `POST /tracking/ingest` موثّق بـ **HMAC لكل جهاز**. المسارات: status/assets/
  devices/geofences + enable/disable per asset. تطبيق السائق (مؤجَّل) يستهلك نفس واجهة المزوّد لاحقًا.
- **موديول «إدارة التتبّع» في القائمة الجانبية:** التتبّع مفصول الآن كقسم مستقل في الشريط الجانبي بعنوان
  **«إدارة التتبّع»** يضمّ أربع صفحات مرتّبة حسب الأهمية:
  1. **خريطة التتبّع** (`/tracking`) — لوحة العمليات الحيّة (أدناه).
  2. **الأجهزة** (`/tracking/devices`) — تسجيل أجهزة GPS + جدول الأصول المتتبَّعة (مع إيقاف التتبّع).
  3. **الأسيجة الجغرافية** (`/tracking/geofences`) — تعريف المناطق الدائرية.
  4. **الاشتراك والربط** (`/tracking/subscription`) — شريط الحالة + شراء/تمديد الإضافة + طلب الربط مع «وصل».
  الصفحات الثلاث الأولى تعرض تنبيهًا مع زر إلى صفحة الاشتراك إن لم تكن الإضافة مُفعّلة. المكوّنات المشتركة
  في `pages/tracking/cards.tsx`. عرض القائمة صار مبنيًّا على البيانات (`SECTIONS`) ويُخفي أي قسم فارغ، وعنصر
  «خريطة التتبّع» يستخدم `end` لمطابقة المسار تمامًا حتى لا يبقى نشطًا على الصفحات الفرعية.
- **لوحة عمليات حيّة مطابقة لتصميم Tookan (صفحة «خريطة التتبّع»):**
  **خريطة Leaflet داكنة (CartoDB dark + OSM، بلا مفتاح API**؛ يمكن استبدالها بـ Google لتطبيق
  السائق) **تملأ كامل الصفحة** (full-bleed: تلغي حشوة `<main>` وترتفع `calc(100vh-4rem)` بلا أي سكرول، ولا
  يوجد عنوان صفحة)، وعليها علامات المركبات (نشطة=أزرق · متاحة=أخضر · غير متصلة=رمادي) والمهام النشطة، مع
  **شريطين جانبيين قابلين للطيّ بسهمين**: **يسارًا «المهام»** (تبويبان: في الانتظار = طلبات معدات PENDING ·
  نشطة = عقود ACTIVE/EXTENDED) و**يمينًا «المركبات»** بثلاثة تبويبات حسب الحالة:
  **نشطة** (متصلة + على مهمة) · **متاحة** (متصلة + شاغرة جاهزة) · **غير متصلة** (لا إشارة حديثة = خارج
  الخدمة/صيانة). **التتبّع للمركبة لا للسائق** (جهاز GPS على الأصل)، فالكيان المُتتبَّع هو **الأصل المُفعّل
  عليه التتبّع** (`trackingEnabled`): الحالة = `!online ? OFFLINE : (عقد نشط ? ACTIVE : AVAILABLE)`،
  وonline يعني آخر إشارة ≤15د. النقر على مهمة/مركبة يُمركز الخريطة؛ تحديث تلقائي.
  المصدر: `GET /tracking/console` (مركبات+مهام+إحداثيات+منطقة).
  - **فلتر المدينة:** قائمة منسدلة أعلى الخريطة (**«كل المدن»** + المدن المُعرَّفة في `LOOKUP_TYPES.REGION`)
    تُصفّي الخريطة والشريطين والعدّادات معًا حسب `region` (حقل `Asset.region`)، وتُعيد تمركز الخريطة على
    أسطول المدينة المختارة. (التصفية في الواجهة على بيانات الكونسول؛ المهام تُصفّى بمنطقة أصل العقد/الأصل المحجوز.)
  - **عدّادات صغيرة** (chips) عائمة أعلى منتصف الخريطة بلا شريط: نشطة · متاحة · غير متصلة · في الانتظار · نشطة(مهام).
  - **بطاقات مفصّلة (تصميم كرت مستطيل):** كرت المركبة = أيقونة شاحنة + الكود + الماركة/الموديل + شارة الحالة +
    **المشروع** + **السائق** المُسنَد (أو «غير مُسند») + **آخر ظهور**؛ كرت المهمة النشطة = رقم التعميد + كود
    المعدّة + **المشروع** + **اسم السائق** + منذ البدء؛ كرت المهمة المعلّقة = المرجع + **المشروع** +
    **النوع المطلوب** + **منذ الطلب**. النوع المشترك `ConsoleVehicle` (id الأصل + code + name + driverName +
    projectName + taskRef + since + إحداثيات + status) و`ConsoleTask`؛ يُبنى عبر `asset(trackingEnabled)` ←
    `vehicle.currentDriver` + ربط العقد النشط بالأصل — دون استعلامات إضافية ثقيلة.

### 3.8 طلب الربط مع «وصل» (Phase 7)
- نموذج `IntegrationRequest` (type قابل للتوسعة). **لا تفعيل تلقائي**: الشركة تطلب → فريق المنصّة يدير
  دورة الحالة `REQUESTED → UNDER_REVIEW → IN_SETUP → ACTIVE → REJECTED/CANCELLED` يدويًا. عند ACTIVE
  يُتاح WASL كمزوّد موقع. المسارات: `POST/GET /integrations` + `/integrations/:id/cancel` (الشركة) ·
  `GET /integrations/all` + `PATCH /integrations/:id/status` (المنصّة). الواجهة: كرت في صفحة التتبّع + كرت إدارة في لوحة المنصّة.

### 3.9 الأدوار والصلاحيات معزولة لكل شركة
- `Role` و`RolePermission` يحملان `tenantId` (ضمن `TENANT_MODELS`)، واسم الدور فريد
  **داخل الشركة** (`@@unique([tenantId, name])`).
- **كل شركة تشكّل أدوارها وصلاحياتها الخاصة بمعزل تام** — إنشاء دور مخصّص أو تعديل
  صلاحيات دور في شركة لا يؤثر على غيرها.
- التزويد عبر `provisionTenantRoles()` المشترك: يُستنسخ كتالوج الأدوار القياسي لكل
  شركة جديدة (في seed، التسجيل الذاتي، إنشاء شركة من المنصّة).
- كتالوج الصلاحيات (`Permission`) يبقى **عامًا** (51 صلاحية = قدرات الكود الفعلية)؛
  الشركات تركّبها في أدوارها فقط.

### 3.10 تحسينات تنظيم الواجهة (تبويبات)
- **الوحدات التنظيمية (`/org-units`)** — تبويبان عبر مكوّن `TabBar` المشترك:
  **«الشكل الهرمي»** (الشجرة المتداخلة القابلة للإدارة: إضافة/تعديل/حذف) و**«الشكل الهيكلي»**
  (مخطّط تنظيمي top-down بـ CSS صرف — موصلات عبر `::before/::after`، صناديق فيها شارة النوع +
  الاسم + اسم المدير، نقر الصندوق يفتح التعديل) ليعطي تصوّرًا واضحًا لهيكلة الشركة.
- **إعدادات النظام (`/system-settings`)** — صفحة واحدة بتبويبات، ومحتوى القوائم المرجعية
  **مفصول إلى تبويب لكل نوع** بدل تكديسه:
  - **تبويب لكل `LookupType`** (المناطق/المدن · الماركات المصنّعة · تصنيفات الأصول — تُبنى ديناميكيًا
    من `LOOKUP_TYPES` بأيقونة لكل نوع) عبر `LookupTypeManager({typeKey})` (يُدير نوعًا واحدًا ويتشارك
    استعلام `['lookups-manage']` المخزَّن مؤقتًا).
  - **«موديلات المركبات» تحت تبويب «الماركات المصنّعة»** — الموديلات مرتبطة بالماركة (Brand→Model) وهي
    مفهوم خاص بالمركبات؛ لذا `ModelsSection` تُعرَض أسفل قائمة الماركات لا في الكتالوج.
  - **«أنواع المعدّات»** (`AssetTypesManager`).
  - **«تصنيف الأصول»** = `ClassesSection` فقط (الأصناف `AssetClass` بملفات الحقول VEHICLE/EQUIPMENT/GENERIC) —
    **هذا هو «الكتالوج»: تصنيف الأصول بلا موديلات**.
  المساران القديمان `/catalog` و`/reference-lists` يُعاد توجيههما إلى `/system-settings`. القائمة الجانبية
  تعرض «إعدادات النظام» بدل العنصرين المنفصلين. (تبويبا أنواع المعدّات وتصنيف الأصول والموديلات حسب صلاحية
  `asset_types`.)

## 4) قابلية التخصيص (إجابة سؤال «هل المنصّة قابلة للتوسّع/التخصيص؟»)

نعم بالكامل ومعزولًا لكل شركة:
- **مستخدمون:** إنشاء + إسناد أدوار، كل دور بنطاق وحدة اختياري؛ **البوابة تتحدّد تلقائيًا
  من الصلاحيات** (القائمة الجانبية والمسارات محميّة بـ `hasPermission`).
- **هيكل إداري:** إنشاء وحدات `DIVISION / DEPARTMENT / PROJECT` متداخلة (`parentId`) بمدير.
- **أدوار/صلاحيات:** إنشاء أدوار خاصة + مصفوفة صلاحيات قابلة للتحرير (51 صلاحية).
- **حماية متعدّدة الطبقات:** RBAC (Guard) + Row-Level (نطاق الوحدة عبر `SCOPED_ROLES`)
  + Field-Level (إخفاء الحقول المالية بـ `finance.read`).

## 5) الميزات الوظيفية (الأساس المبني)

- **الأصول + الكتالوج:** أصناف قابلة للضبط (`AssetClass`) + كتالوج ماركة←موديل (`Model`)
  + حقول مخصّصة لكل نوع (`AssetType.customFields`) + لون. كرت أصل يتكيّف بالنوع
  (مركبة: ماركة/موديل/هيكل/لون؛ معدة: حقول الأدمن المخصّصة). تبويبات وعدّادات غنية.
- **تأكيد الجاهزية (Commissioning):** كل أصل جديد بحالة COMMISSIONING حتى تأكيد قائمة
  السلامة/التركيب/الامتثال → AVAILABLE.
- **النقليات/التأجير (نقطة الحسم #1):** طلب مدير المشروع → **النقليات تُسند الأصل وتشغّله
  في خطوة واحدة** → الأصل «قيد التشغيل لدى المشروع» مع ذكر المشروع. عدّاد أيام + تمديد/تسليم.
- **الاستلام/التسليم (#4):** قوائم فحص + توقيع داخلي موثّق (اسم/دور/ختم زمني/IP/صور) +
  مقارنة حالة (Diff). «لا تسليم بلا فحص».
- **الصيانة + الوقائية (#2):** أوامر + كرت + فواتير + إغلاق وترحيل تكلفة؛ **عدّادات/
  كيلومترات** (`MeterReading`) + **جداول وقائية** (`MaintenancePlan`) + **لوحة امتثال**
  (انتهاء الاستمارة/الفحص/التأمين).
- **المالية/الإهلاك (#3 — هجين):** قيمة دفترية محسوبة آليًا + تجاوز يدوي + TCO (إهلاك +
  صيانة + إيجار) — تظهر لمن يملك `finance.read` فقط.
- **البيع (#5):** اعتماد واحد بفصل مهام (المقترِح ≠ المعتمِد) + ربح/خسارة مقابل الدفترية.
- **الاستحواذ الخارجي (#6):** متحمّل صيانة/تأمين لكل عقد → احتساب OpEx/TCO.
- **السائقون:** سجلات + إسناد مركبة (سائق واحد لكل مركبة).
- **المؤشرات التنفيذية:** تُحتسب لحظيًا من بيانات النظام (تعكس تقرير الجرد التنفيذي).
- **التنبيهات:** انتهاء العقود/الوثائق + قرارات (تكلفة صيانة عالية / نقص توريد) + ملخّص
  بريدي مجدوَل (Resend، بتعطيل لطيف).
- **الحوكمة:** أكواد مرجعية فريدة (SALE-/WO-/LEASE-/AUTH-/TNT-/USR-)، سجل تدقيق
  append-only، آلة حالة محكومة، حذف منطقي.

## ملحق: تفعيل RLS

العزل الحالي مفروض في **طبقة التطبيق** (وسيط Prisma) ومُختبَر 7/7. تفعيل PostgreSQL **RLS**
كطبقة دفاع ثانية مؤجَّل عمدًا لأن الاتصال الحالي بدور `postgres` (superuser **يتجاوز RLS**).
خطوات التفعيل عند الجاهزية (يُنفَّذ في بيئة الإنتاج/قاعدة مُدارة):

1. **دور تطبيق مقيّد:** `CREATE ROLE nxlam_app NOINHERIT LOGIN PASSWORD …;` + `GRANT`
   على الجداول، **بدون** `BYPASSRLS` و**ليس** مالك الجداول؛ وجّه `DATABASE_URL` إليه.
2. **السياسات:** لكل جدول في `TENANT_MODELS`:
   `ALTER TABLE "X" ENABLE ROW LEVEL SECURITY;`
   `CREATE POLICY tenant_isolation ON "X" USING ("tenantId" = current_setting('app.current_tenant', true) OR current_setting('app.current_tenant', true) IS NULL);`
   (شرط `IS NULL` يبقي seed/المنصّة/المسارات العامة تعمل بلا سياق.)
3. **ضبط السياق لكل طلب:** امتداد Prisma (`$extends`) يضبط `SELECT set_config('app.current_tenant', <tenantId>, true)`
   داخل معاملة قبل الاستعلام عندما يكون `currentTenantId()` موجودًا — مع تحويل مواضع
   `$transaction([...])` الستّ إلى معاملات تفاعلية تضبط الـ GUC أولًا.
4. **اختبار:** تأكيد أن RLS يمنع التسرّب حتى لو عُطِّل وسيط التطبيق (اختبار سلبي مباشر على القاعدة).

> الأثر على الكود الحالي صفر اليوم؛ التفعيل تغيير بيئة + امتداد Prisma، دون لمس منطق الأعمال.
