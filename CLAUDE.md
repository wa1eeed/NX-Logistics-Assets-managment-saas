## 1) الستاك المعتمد

- **Monorepo:** pnpm workspaces (`apps/api`, `apps/web`, `packages/shared`، ولاحقاً `apps/driver`).
- **Backend:** NestJS + TypeScript (موديولز وكلاسز بطبيعتها: Modules / Controllers / Services / Guards / DTOs).
- **ORM/DB:** Prisma + PostgreSQL (المخطط جاهز في `prisma/schema.prisma`).
- **Frontend:** React + Vite + TypeScript، متعدّد البوابات، **ثنائي اللغة عربي/إنجليزي مع RTL**.
- **Auth:** JWT (access + refresh) مع RBAC على مستوى الـ Guard + scoping على مستوى الاستعلام.
- **التخزين:** Cloudflare R2 (متوافق S3) + Cloudflare CDN للروابط العامة.
- **البريد/الإشعارات:** Resend.com (transactional).
- **تتبع الأخطاء:** Sentry (api + web).
- **النشر:** Docker → Coolify على VPS (بيئة staging).
- **الخرائط (لاحقاً):** Google Maps API لتطبيق السائقين — لا تبنِه الآن، لكن لا تغلق الطريق أمامه (انظر §6).

> إن رأيت بديلاً أفضل لأي خيار، **اطرحه عليّ كاقتراح ولا تبدّله من تلقاء نفسك**.

## 2) مبادئ البناء (إلزامية)

- **TypeScript strict** في كل مكان، لا `any` إلا بمبرر موثّق.
- **معماري موديولز/كلاسز:** كل مجال (Domain) موديول مستقل: `assets`, `asset-types`, `vehicles`, `drivers`, `documents`, `rentals` (الطلب/التعميد/العقد), `handover` (الاستلام/التسليم), `maintenance`, `org-units` (مشاريع/إدارات), `disposal` (البيع), `acquisition` (الاستحواذ الخارجي), `suppliers`, `rbac` (users/roles/permissions), `audit`, `notifications`, `integrations`.
- **طبقات واضحة:** Controller (HTTP) → Service (منطق) → Repository/Prisma (بيانات) → DTO (تحقّق عبر class-validator).
- **State Machine للأصل** في خدمة مركزية (`AssetStatusService`): لا تُغيَّر حالة الأصل إلا عبرها، مع تحقّق الانتقالات المسموحة (راجع §4 في التصوّر).
- **سجل تدقيق (AuditLog)** عبر Interceptor عام يلتقط كل عملية كتابة (من/ماذا/قبل/بعد/متى).
- **لا أسرار في الكود:** كل المفاتيح من `.env` (انظر `.env.example`). شغّل التطبيق في dev حتى لو لم تُضبط مفاتيح R2/Resend/Sentry (تعطيل لطيف / Feature flags).
- **الترحيلات (Migrations) عبر Prisma** فقط، لا تعديل يدوي على القاعدة.
- **i18n + RTL** في الواجهة من البداية (مفاتيح ترجمة، اتجاه الصفحة حسب اللغة).

## 3) ترتيب البناء (ابنِ على مراحل، لا دفعة واحدة)

ابدأ بالمرحلة 1 (MVP) واطلب مراجعتي قبل الانتقال:

**المرحلة 1 (MVP):**
1. تهيئة المونوريبو + Prisma + اتصال Postgres (عبر `docker-compose.yml`) + تشغيل أول Migration.
2. موديول `rbac` (مستخدمون/أدوار/صلاحيات) + Auth + Guards + scoping.
3. موديول `assets` + `asset-types` + `vehicles` + State Machine + `documents` (رفع إلى R2).
4. موديول `org-units` (مشاريع) + `rentals` (طلب→تعميد→عقد→عهدة) + عدّاد الأيام.
5. موديول `maintenance` (أمر صيانة + كرت + رفع فواتير) + `audit`.
6. لوحة السوبر أدمن (إدارة المستخدمين/الصلاحيات/البيانات المرجعية).

**المرحلة 2:** فحص الاستلام/التسليم المقارن (Condition Diff) + التنبيهات (Resend) + لوحات المؤشرات.

**المرحلة 3:** `disposal` (البيع وقسم المعروضة للبيع) + `acquisition` (شراء/استئجار خارجي بتمييز التكاليف) + توسعة للإدارات + تقارير حوكمة.

## 4) 🔴 نقاط الحسم — اسألني عند الوصول إليها

**لا تفترض هذه القرارات. عند بلوغ النقطة المعنية في التنفيذ، توقّف واطرح السؤال عليّ ثم تابع.** التفاصيل في `docs/05_decision_points.md`:

1. **عند بناء اعتماد التعميد:** اعتماد واحد أم سلسلة اعتمادات؟
2. **عند بناء الصيانة الوقائية:** هل أربط عدّاد التشغيل/الكيلومترات بجدولة الصيانة؟
3. **عند بناء التاب المالي/الإهلاك:** يُحتسب داخل المنصة أم يُجلب من نظام محاسبي خارجي (تكامل)؟
4. **عند بناء الاستلام/التسليم:** توقيع إلكتروني رسمي أم تأكيد داخلي؟
5. **عند بناء موديول البيع:** اعتماد واحد (مدير الأصول) أم اعتماد مالي إضافي قبل الخصم؟
6. **عند بناء الاستحواذ الخارجي:** من يتحمّل صيانة/تأمين الأصل المستأجَر (الشركة أم المورّد)؟ — يحدّد حساب OpEx/TCO.

## 5) حماية البيانات (متطلبات غير قابلة للتفاوض)

- الصلاحيات تُطبَّق **على مستوى الاستعلام** لا الواجهة فقط (Row-Level scoping عبر org_unit/الدور).
- الحقول الحسّاسة (القيمة الدفترية، قيمة العقد، أجرة المورّد) تُخفى بـ DTO حسب الصلاحية (Field-Level).
- ملفات R2 في bucket **خاص**، والوصول عبر **Presigned URLs قصيرة الأمد** فقط.
- كلمات المرور: Argon2/bcrypt. نقل مشفّر (TLS). Helmet + Rate limiting + CORS مضبوط.
- `AuditLog` غير قابل للتعديل (append-only).
- حذف منطقي (`deletedAt`) لا حذف فيزيائي للكيانات المحورية.

---

## 6) اعتبارات تطبيق السائقين (مرحلة لاحقة — لا تبنِه الآن)

- ضع الأنواع المشتركة (Asset/Contract/Driver DTOs) في `packages/shared` ليعاد استخدامها في `apps/driver` (React Native) لاحقاً.
- اجعل API الـ rentals/handover **agnostic** عن العميل (REST نظيف + JWT) ليستهلكه تطبيق السائق.
- احجز في المخطط ربط السائق بالعقد/الأصل وقت التشغيل (موجود)، وأبقِ متغيّر `GOOGLE_MAPS_API_KEY` في `.env.example` كحجز مكان.

## 7) التشغيل والنشر

- **dev:** `docker-compose up -d` (Postgres) ثم `pnpm install` و `pnpm --filter api prisma migrate dev` ثم `pnpm dev`.
- **staging:** صور Docker لكل من api/web (انظر `deploy/`) تُنشر عبر Coolify على VPS، والأسرار تُدار من Coolify secrets لا من الريبو.
