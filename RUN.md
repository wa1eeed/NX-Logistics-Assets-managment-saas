# NX-LAM — تشغيل بيئة التطوير (dev)

> **المنصّة الآن SaaS متعدّدة المستأجرين.** للمرجع الشامل لما نُفّذ راجع
> [`docs/06_implemented_state.md`](docs/06_implemented_state.md). طبقة الـ SaaS موثّقة في
> قسم «طبقة الـ SaaS» أسفل هذا الملف.

> الشرائح المُسلّمة:
> 1. **لوحة السوبر أدمن** — المستخدمون/الأدوار/الصلاحيات + البيانات المرجعية + عتبات التنبيه + سجل التدقيق.
> 2. **لوحة إدارة الأصول** — تسجيل الأصول + ملف الأصل بتابات (عام/مركبة/مستندات/مالية/خط زمني) + رفع المستندات + State Machine للحالة.
> 3. **لوحة النقليات والتأجير** — طلب ← اعتماد (حجز أصل) ← تعميد/عقد ← عهدة + عدّاد الأيام + تمديد/تسليم.
> 4. **بوابة مدراء المشاريع** — صفحة «عهدتي» بنطاق المشروع: ملخص (بالعهدة/قارب الانتهاء/متأخر) + بطاقات العهدة مع تنبيه الانتهاء + طلب معدة + تمديد/تسليم.
> 5. **بوابة الصيانة** — أوامر صيانة (تصحيحية/وقائية) + كرت الصيانة (أعمال/قطع غيار/فنّي/ساعات) + رفع الفواتير + الإغلاق وترحيل التكلفة + State Machine للأصل.
> 6. **اللوحة التنفيذية / مؤشرات الأسطول** — تعيد إنتاج مؤشرات تقرير جرد الأصول التنفيذي **لحظياً** من بيانات النظام.
> 7. **الاستلام/التسليم وفحص الحالة** — قوائم فحص + توقيع داخلي موثّق + مقارنة الحالة (Diff) وعدّاد العدّاد.
> 8. **البيع والتصفية** — أوامر بيع باعتماد واحد (فصل مهام) + ربح/خسارة مقابل القيمة الدفترية.
> 9. **الموردون والاستئجار الخارجي** — متحمّل الصيانة/التأمين لكل عقد (OpEx/TCO).
> 10. **السائقون** · 11. **التنبيهات** (Resend بتعطيل لطيف) · 12. **بنية الإدارات/الوحدات** + أدوار الوحدات.
> الواجهة أُعيد تصميمها بالكامل على **Tailwind + Shadcn UI + Framer Motion** مع وضع ليلي.
> **الحوكمة:** أكواد مرجعية فريدة لكل معاملة (SALE-/WO-/LEASE-/AUTH-)، سجل تدقيق غير قابل للتعديل بختم زمني وفاعل لكل كتابة، آلة حالة محكومة، فصل مهام، وإخفاء الحقول المالية حسب الدور.

## ما الذي تتحقق منه في هذه الشريحة
1. تسجيل الدخول بحساب السوبر أدمن.
2. **لوحة التحكم**: عدّادات النظام.
3. **المستخدمون**: إنشاء/تعديل/تعطيل + تعيين أدوار مع نطاق (وحدة تنظيمية).
4. **الأدوار والصلاحيات**: مصفوفة صلاحيات قابلة للتحرير لكل دور.
5. **الوحدات التنظيمية**: شجرة قطاع/إدارة/مشروع مع CRUD.
6. **أنواع المعدات**: CRUD.
7. **الإعدادات/العتبات**: تحرير عتبات التنبيه واللغة الافتراضية.
8. **سجل التدقيق**: يسجّل كل عمليات الكتابة (append-only) مع تصفية.
9. الصلاحيات مطبّقة **على مستوى الاستعلام** (دور بلا صلاحية ⇒ 403، بلا توكن ⇒ 401)، والنطاق يُحسب هرمياً.

## الشريحة 2 — إدارة الأصول
1. **قائمة الأصول**: فلاتر (الحالة/الملكية/النوع/بحث) + شارات حالة + تسجيل أصل جديد.
2. **ملف الأصل** بتابات: عام، المركبة، المستندات، المالية، الخط الزمني.
3. **المالية (هجين — نقطة الحسم #3)**: المنصة تحسب القيمة الدفترية آلياً (قسط ثابت من سعر الشراء × النسبة × العمر)، ويمكن إدخال **قيمة يدوية تتجاوزها**. الحقول المالية تظهر فقط لمن يملك `finance.read`.
4. **State Machine**: تغيير الحالة عبر انتقالات مسموحة فقط (الأصل المستأجَر خارجياً لا يُباع؛ المملوك يُستبعَد عبر البيع). انتقال غير مسموح ⇒ 400.
5. **المستندات**: رفع إلى R2 — وفي dev (بلا مفاتيح R2) **تخزين محلي بديل** تحت `.uploads/` مع روابط تنزيل موقّعة قصيرة الأمد. الملفات المرفوعة تظهر في تاب المستندات مع تنزيل/حذف.
6. الخط الزمني يعرض أحداث الأصل من سجل التدقيق (بما فيها تغييرات الحالة).

## الشريحة 3 — النقليات والتأجير
1. **الطلبات**: مدير المشروع يطلب معدة (نوع + فترة + وحدة تنظيمية ضمن نطاقه) ← النقليات **تعتمد** (تختار أصلاً متاحاً ← يصير محجوزاً RESERVED) أو ترفض ← **تُصدر التعميد والعقد** (رقم تعميد آلي AUTH-YYYY-NNNN) ← الأصل IN_DUTY + عهدة على الوحدة + الطلب FULFILLED.
2. **العقود**: قائمة بالعقود السارية مع **عدّاد الأيام المتبقية** (أحمر متأخر / أصفر ≤14 يوماً) + **تمديد** + **تسليم** (الأصل يعود AVAILABLE أو FOR_SALE إن كان مؤشّراً «لا يُعاد تأجيره»).
3. **نقطة الحسم #1 = اعتماد واحد (النقليات)** — خطوة موافقة واحدة قبل التعميد.
4. الصلاحيات: `rentals.request` (طلب) · `rentals.approve` (اعتماد/رفض) · `rentals.contract` (تعميد) · `rentals.extend` · `rentals.return`. **القيمة المحاسبية (internalRate)** تظهر فقط لمن يملك `finance.read`. **النطاق**: مدير المشروع يرى/يطلب لوحداته فقط.

## الشريحة 4 — بوابة مدراء المشاريع
1. صفحة **«عهدتي»** (`/custody`): بطاقات ملخص (بالعهدة / قارب الانتهاء / متأخر) + بطاقات العهدة لكل أصل مع **عدّاد الأيام وتنبيه قرب الانتهاء** (بحسب عتبة `alerts.contractExpiryDays`).
2. مدير المشروع: **طلب معدة** + **تمديد** + **تسليم** مباشرة من البطاقة.
3. **حماية البيانات مُتحقَّق منها**: مدير المشروع يرى عهدة/طلبات/عقود **مشروعه فقط**؛ الشريط الجانبي يُخفي صفحات الإدارة؛ محاولة فتح `/users` تُعيد توجيهه؛ واستدعاء API إداري يُرجع 403؛ والقيمة المحاسبية مخفية عنه. لا نقاط حسم جديدة في هذه الشريحة.

## الشريحة 5 — بوابة الصيانة
1. **أوامر الصيانة** (`/maintenance`): إنشاء أمر (الأصل/المصدر/النوع تصحيحية·وقائية/الأولوية/الوصف) → **بدء** (الأصل يصبح UNDER_MAINTENANCE) → تعبئة **كرت الصيانة** (أعمال + قطع غيار بكمية وتكلفة + الفنّي + ساعات العمل) → **رفع الفواتير/كشوف الفحص** → **إغلاق** (إدخال التكلفة الإجمالية، الأصل يعود AVAILABLE).
2. الصلاحيات: `maintenance.create` (فتح/بدء/إلغاء) · `maintenance.card` (الكرت) · `maintenance.close` (الإغلاق) · `documents.upload` (الفواتير). انتقالات الحالة محكومة عبر State Machine (إغلاق أمر مغلق ⇒ 400).
3. **نقطة الحسم #2 = لا ربط بالعدّاد الآن** — النوع وقائية/تصحيحية كتصنيف فقط، دون عدّاد تشغيل/كيلومترات أو جدولة آلية.
4. تاب **«الصيانة»** في ملف الأصل يعرض أوامر صيانة ذلك الأصل (يظهر لمن يملك `maintenance.read`). الفواتير تُخزَّن عبر نفس آلية R2/المحلي.

## الشريحة 6 — اللوحة التنفيذية / مؤشرات الأسطول (`/fleet-kpis`)
تعيد إنتاج **نفس مؤشرات تقرير جرد الأصول التنفيذي** محسوبة **لحظياً** من بيانات الأصول والعمليات (تحديث تلقائي كل 60 ثانية وعند كل تنقّل):
1. **بطاقات:** إجمالي الأصول · عاملة · **نسبة الجاهزية %** · متوقفة · متوقفة للإصلاح · أصول تتطلب قراراً.
2. **نطاق التغطية:** عدد المناطق · المواقع · أنواع المعدات · الماركات.
3. **دونات** توزيع الحالة التشغيلية · **جدول جاهزية المناطق** بأشرطة · **التوزيع حسب التصنيف** · **الهيكل العمري** · **أبرز الماركات** · **أعلى أنواع المعدات** · **تركّز الأصول غير العاملة** · **جودة البيانات** (بلا لوحة/بلا سنة صنع).
4. أُضيفت حقول `manufacturer`/`region`/`siteName` للأصل و`category` لنوع المعدة، وبيانات أولية موزّعة على 8 مناطق لتغذية المؤشرات.
5. الصلاحية `kpis.read`. المصدر: `GET /api/kpis/fleet`. أي تغيير حالة أصل ينعكس فوراً على المؤشرات.

> **ملاحظة:** الأرقام تعكس بيانات النظام (الأصول المُدخلة)، لا أرقام التقرير الورقي — وهذا هو المطلوب: نفس المؤشرات تُحتسب لحظياً وتكبر مع إدخال الأصول الفعلية.

## القوائم المرجعية القابلة للضبط (`/reference-lists`)
حقول الـ dropdown في المنصة تُدار من السوبر أدمن (نموذج `Lookup` عام):
1. **المناطق/المدن** · **الماركات المصنّعة** · **التصنيفات** — لكل عنصر تسمية إنجليزي/عربي + قيمة مخزّنة + ترتيب + تفعيل/تعطيل + CRUD.
2. تُغذّي قوائم النماذج مباشرة: **المنطقة والماركة في تسجيل/تعديل الأصل**، و**تصنيف نوع المعدة**. إضافة مدينة جديدة من السوبر أدمن تظهر فوراً في القائمة المنسدلة.
3. القراءة متاحة لأي مستخدم (لتعبئة القوائم)، أما **الإدارة (إضافة/تعديل/حذف) فمحصورة بـ `settings.manage`** (السوبر أدمن) — تحقّقت: مدير المشروع يقرأ (200) ولا يضيف (403).
4. توسيع القوائم لاحقاً (أنواع المستندات، الأولويات…) = إضافة نوع في `LOOKUP_TYPES` فقط.

## تأكيد الجاهزية عند تسجيل الأصل (Commissioning)
- أي أصل جديد يُدرَج بحالة **COMMISSIONING** (ليس متاحاً للتشغيل) — هذا الافتراضي في المخطط، لا يمكن تخطّيه من الـ API.
- في صفحة الأصل تظهر لوحة **«تأكيد الجاهزية»** بثلاث مجموعات: نقاط سلامة + تركيب أجهزة + امتثال. لا يتفعّل زر التأكيد إلا بعد تحديد كل البنود **المطلوبة** (`READINESS_REQUIRED_KEYS`).
- عند التأكيد: تُحفظ قائمة الفحص + `commissionedAt` + `commissionedBy`، وتنتقل الحالة عبر آلة الحالات إلى **AVAILABLE** (مع قيد STATUS_CHANGE في سجل التدقيق). المسار: `POST /assets/:id/commission`. بيانات الـ seed تتضمّن TRK-005 بانتظار التأكيد.

## استكمالات الـ Workflow (الحوكمة الكاملة)
1. **بوابة الاستلام/التسليم:** لا يمكن إرجاع عقد (`POST /rentals/contracts/:id/return`) دون تسجيل **فحص إرجاع (RETURN)** — و الإرجاع نفسه يتطلّب فحص استلام (RECEIPT) مسبقاً، فيُوثَّق الطرفان («لا تسليم بلا فحص»).
2. **TCO على الأصل:** تبويب المالية يعرض **إجمالي تكلفة التملّك** = الإهلاك المتراكم + إجمالي الصيانة (أوامر مغلقة) + الإيجار المستحق، ونسبة الصيانة/القيمة الدفترية (تُبرَز حمراء فوق 50%). مرئي لمن يملك `finance.read` فقط.
3. **تنبيهات القرار** (`GET /alerts`): إضافة `MAINTENANCE_COST` (تكلفة صيانة تتجاوز نسبة من القيمة الدفترية → تجديد/بيع) و`SUPPLY_SHORTAGE` (طلب معلّق تجاوز مهلته → إشارة استحواذ). العتبات قابلة للضبط: `alerts.maintenanceCostRatio` و`alerts.requestPendingDays`.
4. **لوحات مؤشرات الإدارات:** صفحة الصيانة فيها شريط (مفتوحة/قيد التنفيذ/مغلقة/التكلفة/MTTR/نسبة الوقائية + أعلى الأصول تكلفة) عبر `GET /kpis/maintenance`؛ وصفحة النقليات فيها شريط (نسبة التشغيل/في الخدمة/متاحة/طلبات معلّقة/معتمدة/عقود نشطة) عبر `GET /kpis/dispatch`.
5. **ملخّص تنبيهات مجدوَل بالبريد:** مُجدوِل ذاتي (بلا تبعيات) يرسل ملخّص التنبيهات للمستخدمين الذين يملكون `kpis.read`؛ معطّل بلطف عند غياب `RESEND_API_KEY` (يكتفي بالسجل). تشغيل يدوي: `POST /alerts/digest/run`. الإعداد: `ALERTS_DIGEST_INTERVAL_MS` و`ALERTS_DIGEST_TO`.
6. **إسناد السائق للمركبة:** من صفحة السائقين زر «إسناد مركبة» يفتح نافذة اختيار مركبة (يعرض الحائز الحالي) — `PATCH /drivers/:id/assign` و`GET /drivers/assignable-vehicles`. السائق يحمل مركبة واحدة كحد أقصى، والإسناد يستبدل أي سائق سابق على نفس المركبة.

## نموذج البوابات والصلاحيات (مراجعة شاملة)

### مبدأ النطاق (Scoping) — مُصحَّح
النطاق الصفّي (Row-Level) يُطبَّق **فقط على الأدوار المستهلِكة للأصول** (مدير المشروع) عبر `SCOPED_ROLES` في `packages/shared/src/roles.ts`. أمّا الإدارات/الوظائف المركزية (إدارة الأصول، النقليات، الصيانة، المعتمِد، التسجيل) فهي **شاملة على مستوى الشركة** لأن وظيفتها رصد/تشغيل كامل الأسطول — حتى لو سُجّلت عضويتها في إدارة معيّنة (تبقى للهيكل التنظيمي فقط). الحساب في `access.service.ts`: من يملك أي دور غير مُقيَّد → نطاق شامل؛ ومن كل أدواره مُقيَّدة → يُقصر على وحداته.

> **قبل الإصلاح** كانت كل الأدوار مُقيَّدة بوحدتها فترى **صفر أصول** (مدير الأصول/النقليات/الصيانة لا يرون الأسطول!). **بعده:** المركزية ترى 21 أصلاً، ومدير المشروع يرى عهدة مشروعه فقط (2).

### فصل المهام (SoD) — مُطبَّق على مستوى الصلاحية
- **النقليات (DISPATCH):** تُصدر التعميد/العقد والتسليم — **لا تعتمد** (أُزيلت `rentals.approve`).
- **المعتمِد (UNIT_APPROVER):** يعتمد الطلبات/البيع فقط — **لا يُصدر عقوداً** (أُزيلت `rentals.contract`) ولا يُغلق صيانة.
- **البيع:** نظام يمنع اعتماد الشخص لاقتراحه (`proposedBy === user` → رفض) في `disposal.service`.

### أقل صلاحية (Least Privilege) — تنظيف البوابات
- **التسجيل (UNIT_OPERATOR):** بوابة تسجيل/توثيق فقط — أُزيلت `rentals.*` و`maintenance.*`. القائمة: لوحة، الأصول، السائقون، أنواع المعدات.
- **مدير المشروع (PROJECT_MANAGER):** أُزيلت `kpis.read` لمنع تسرّب اللوحة التنفيذية والتنبيهات على مستوى الشركة. القائمة: لوحة، الأصول (نطاقه)، التأجير (نطاقه)، عهدتي، أنواع المعدات.
- **بطاقات لوحة التحكم** تُفلتر حسب الصلاحية (مثلاً لا تظهر «المستخدمون/الأدوار» لمدير الإدارة)، وعدّاد الأصول في اللوحة **مُقيَّد بنطاق المستخدم**.

# طبقة الـ SaaS (متعدّدة المستأجرين)

> المرجع الكامل: [`docs/06_implemented_state.md`](docs/06_implemented_state.md).

## أ) العزل متعدّد المستأجرين
كل كيان جذري يحمل `tenantId`، ويُفرَض العزل **على مستوى الاستعلام** عبر وسيط Prisma
(`TENANT_MODELS`) — كل شركة ترى بياناتها فقط (NX ترى 1947 أصلاً، شركة جديدة ترى 0).
السياق يُضبط من الـ JWT (`TenantContextInterceptor`).

## ب) لوحة مشغّل المنصّة (`/platform`)
- هوية المشغّل في **جدول `PlatformAdmin` مستقل** (ليس مستخدم شركة) لعزل أثر الاختراق.
- عرض كل الشركات + الاستهلاك + MRR تقديري · إنشاء/تعليق شركة · تعديل الاشتراك ·
  **انتحال (impersonate)** للدعم · نشاط عابر للشركات · إعدادات **التخزين المشترك** و**بوابة Tap**.
- لا يملك أي صلاحيات بيانات شركة (يصل لبياناتها بالانتحال فقط).

## ج) الاشتراكات والاستحقاقات (Entitlements)
- لكل شركة باقة بسقوف (مستخدمون/تخزين) وموديولات مفعّلة. حُرّاس **403** عند تجاوز
  السقف أو الوصول لموديول غير مفعّل. باقات جاهزة STARTER/STANDARD/ENTERPRISE.

## د) الفوترة الذاتية + المحفظة (`/billing`)
- محفظة مدفوعة مسبقاً · شراء مقاعد إضافية (يرفع حد المستخدمين) · تفعيل إضافات موديولات ·
  سجل حركات. الدفع من المحفظة أو **بالبطاقة** عبر Tap.

## هـ) بوابة الدفع — Tap
- حساب تاجر واحد للمنصّة، يُضبط من **لوحة المنصّة** (المفتاح السرّي مُقنّع، لا يُعاد للواجهة).
- **Hosted Checkout** → عودة `/billing/return` + **Webhook** → تحقّق من Tap → **تطبيق مرة واحدة**.
- **Sandbox** عند غياب مفتاح حقيقي (للتجربة في dev)؛ يُرفض تلقائيًا في وجود مفتاح مباشر.
- الإعداد فعليًا من الواجهة؛ env: `WEB_PUBLIC_URL` + `TAP_SECRET_KEY/TAP_PUBLIC_KEY` (احتياطية).

## و) التسجيل الذاتي للشركات (`/register`)
- صفحة عامة: الشركة تسجّل نفسها → شركة معزولة (TRIAL) + اشتراك + أول SUPER_ADMIN + دخول تلقائي.

## ز) الهوية البصرية لكل شركة (الإعدادات)
- اسم معروض + لون علامة + شعار → تُلوّن الواجهة كاملة (الشعار/الاسم/`--primary`). رفع الشعار ≤2MB.

## ح) الأدوار والصلاحيات معزولة لكل شركة
- `Role`/`RolePermission` يحملان `tenantId`؛ كل شركة تنشئ أدوارها وتفصّل صلاحياتها بمعزل تام
  (تعديل دور في شركة لا يؤثر على غيرها). كتالوج الصلاحيات (51) عام؛ الشركات تركّبها في أدوارها.
- كل شركة جديدة تُزوَّد بمجموعة الأدوار الافتراضية تلقائيًا (`provisionTenantRoles`).

## ط) الأسطول الحقيقي + الحقول المخصّصة
- بيانات الشركة الأولى = **1947 أصلاً حقيقيًا** مستوردة من ملف العميل. كرت الأصل يتكيّف
  بالنوع (مركبة: ماركة/موديل/هيكل/لون؛ معدة: حقول مخصّصة يضبطها الأدمن لكل نوع). كتالوج
  أصناف + ماركة←موديل قابل للضبط من السوبر أدمن.

## ي) الصيانة الوقائية + الامتثال
- عدّادات تشغيل/كيلومترات + جداول وقائية + **لوحة امتثال** (انتهاء الاستمارة/الفحص/التأمين).

> **حسابات إضافية:** يمكن تسجيل شركات جديدة عبر `/register` أو إنشاؤها من لوحة المنصّة؛
> كلٌّ بأدواره وبياناته المعزولة. إعادة البذر (`pnpm --filter api db:seed` أو `prisma migrate reset`)
> تعيد الحالة لشركة NX الأساسية.
